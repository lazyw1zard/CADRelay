# CADRelay Backlog (MVP)

## Sprint 0 - Foundation
- [ ] Определить целевой формат MVP (STEP как основной вход).
- [ ] Зафиксировать архитектурные контракты сервисов (API, worker, storage).
- [ ] Подготовить репозиторный интерфейс для метаданных (без привязки к Firestore).
- [ ] Настроить базовую структуру проекта и правила версионирования API.
- [ ] Описать env-конфиги для локальной разработки.

## Sprint 1 - Upload + Metadata
- [ ] Endpoint загрузки файла и создания `model_version`.
- [ ] Запись оригинала в object storage.
- [ ] Запись метаданных в Firestore.
- [ ] Статусы: `uploaded`, `processing`, `ready`, `failed`.
- [ ] Список моделей/версий в API.

## Sprint 2 - Conversion Worker
- [ ] Очередь задач конвертации.
- [ ] Worker для STEP -> GLB.
- [ ] Сохранение GLB и preview в storage.
- [ ] Обновление статусов и ошибок.
- [ ] Повторная обработка (`retry`) для неуспешных задач.

## Sprint 3 - Viewer + Approval
- [ ] Viewer-страница с загрузкой GLB.
- [ ] Управление камерой (rotate/pan/zoom).
- [ ] Форма решения: Approve/Reject + комментарий.
- [ ] История согласований по версии.
- [ ] Скачивание оригинала и GLB.

## Sprint 4 - Security + Audit
- [ ] Роли и доступ к проектам/моделям.
- [ ] Подписанные URL для загрузки/скачивания.
- [ ] Audit log ключевых операций.
- [ ] Базовые лимиты и валидации входных форматов.

## Definition of Done (MVP)
- [ ] Пользователь загружает STEP, видит статус конвертации, открывает GLB.
- [ ] Клиент принимает решение Approve/Reject с комментарием.
- [ ] Можно скачать исходник и производный GLB.
- [ ] Все ключевые действия логируются.
- [ ] Документация по запуску и архитектуре зафиксирована.

## Post-MVP направления
- [ ] Миграция метаданных на PostgreSQL.
- [ ] Поддержка расширенных форматов через fallback-конвертер.
- [ ] Расширенный viewer: измерения, сечения, BOM.
- [ ] Уведомления и внешние интеграции (PDM/PLM/CRM).
