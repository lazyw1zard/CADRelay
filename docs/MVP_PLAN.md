# CADRelay MVP Plan

## 1. Цель MVP
Собрать минимально рабочий процесс, в котором пользователь:
- загружает CAD-файл (основной формат: STEP + хранение оригинала),
- получает 3D-просмотр в веб-интерфейсе (через GLB),
- оставляет решение по модели (Approve/Reject) с комментарием,
- может скачать исходный файл и подготовленный артефакт.

Критерии успеха MVP:
- поддержка детали и простого сборочного узла,
- асинхронная конвертация со статусами,
- базовые роли (минимум: внутренний пользователь/клиент на просмотр),
- история согласований по версиям.

## 2. Границы MVP
Входит в MVP:
- Upload файла,
- очередь + worker конвертации,
- хранение оригинала и GLB,
- веб-вьюер с базовой навигацией,
- approve/reject + комментарий,
- скачивание файлов.

Не входит в MVP (позже):
- полнофункциональный CAD-редактор в браузере,
- сложная BOM-аналитика,
- продвинутые измерения/PMI/разрезы enterprise-уровня,
- интеграции с PDM/PLM.

## 3. Технологический контур
- Frontend: React + Three.js (или react-three-fiber).
- Backend API: Node.js (Express/Nest) или Python (FastAPI).
- Worker: отдельный сервис (асинхронная конвертация).
- Хранилище файлов: object storage (на MVP можно Cloud Storage).
- База метаданных на MVP: Firestore.

Принцип совместимости с будущим PostgreSQL:
- доменная логика не зависит от конкретной БД,
- доступ к данным через репозитории,
- идентификаторы и версии стабильные, не привязаны к структуре Firestore.

## 4. Форматы и конвертация
Вход:
- приоритетно STEP (AP242),
- оригинал CAD хранится как есть.

Выход:
- GLB для просмотра,
- thumbnail/preview.

Pipeline:
1. Загрузка файла в storage.
2. Создание записи версии модели в БД со статусом `uploaded`.
3. Постановка задачи в очередь.
4. Worker конвертирует в GLB.
5. Запись артефактов в storage, обновление статуса (`ready`/`failed`).
6. Viewer отображает результат и статус.

## 5. Минимальная доменная модель
- project
- model
- model_version
- file_asset
- approval
- user
- audit_event

Минимальные поля версии:
- `model_version_id`
- `source_format`
- `storage_key_original`
- `storage_key_glb`
- `checksum`
- `size_bytes`
- `status`
- `created_by`
- `created_at`

## 6. Нефункциональные требования
- Ограничение размера входного файла на MVP (задаётся конфигом).
- Логи по всем этапам pipeline.
- Идемпотентность worker-задач.
- Базовая безопасность: приватные файлы, скачивание через подписанные URL.

## 7. План миграции в будущем
- Этап 1: Firestore как metadata store.
- Этап 2: репозиторный слой с контрактами, независимыми от Firestore.
- Этап 3: dual-write или one-time migration в PostgreSQL.
- Этап 4: переключение чтения на PostgreSQL.
- Этап 5: вывод Firestore из цепочки.
